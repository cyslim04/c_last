# 管理员成员管理 + 阶段循环式项目流程改造方案

## Summary
- 新增管理员专用的独立“成员管理”页，支持开发者/客户账号的新增、编辑、重置密码、停用/启用，以及“未被项目引用时删除”。
- 将项目流程从“固定 8 步线性链”改为“项目立项 + 开发准备 + 阶段循环 + 最终总审计”。
- 客户确认仍以交付物为主，但确认页必须同时展示同阶段工时历史；管理员在客户通过后执行一次“阶段审计”，项目结束时再做“总审计”。
- 不改 Solidity 合约，阶段审计和总审计继续走现有“演示链上回写”模式，新增链上记录类型即可。

## Key Changes
### 1. 管理员成员管理
- 新增 admin-only 的独立页面 `/members`，在左侧导航加入“成员管理”入口。
- 页面默认分为“开发者 / 客户 / 全部 / 停用”筛选，并提供成员列表与操作区。
- 第一版成员管理做到“完整管理”：
  - 新增成员
  - 编辑资料
  - 重置密码
  - 停用/启用
  - 删除
- 新增成员字段固定为：
  - `username`
  - `password`
  - `name`
  - `role`（仅 `developer | client`）
  - `company`
  - `email`
  - `phone`
  - `bio`
- 创建后 `username` 与 `role` 不允许再修改；编辑只改资料类字段。
- 新增 `users.status`（推荐 `active | inactive`）：
  - `inactive` 账号不可登录
  - `inactive` 账号不可再被分配到新项目
  - 历史项目、审计、链上回写与头像资料保留
- 删除规则按你选的“引用则停用”执行：
  - 未被任何项目引用的成员允许硬删除
  - 已被项目引用的成员不允许硬删除，只能停用
- 项目管理页创建项目时，开发者/客户选择器只展示 `active` 成员。

### 2. 项目流程改为“阶段循环”
- 取消原流程里“先分配开发者与客户”这条独立步骤，合并为“管理员创建项目并确定成员”。
- 钱包绑定保留，但改为“项目启动后的开发准备动作”，不再作为管理员侧单独编排步骤。
- 新的高层流程固定为：
  1. 管理员创建项目并确定成员
  2. 开发者完成首次钱包绑定
  3. 开发者在当前阶段持续登记工时与交付物
  4. 开发者对当前阶段发起链上存证/演示回写
  5. 客户确认当前阶段交付，查看同阶段工时历史
  6. 管理员执行当前阶段审计并记录演示链
  7. 阶段 3-6 可重复
  8. 管理员执行项目总审计并归档
- 流程总览页不再展示单条 8 步线性链，而是改成“阶段循环”模型：
  - 顶部显示项目立项与准备状态
  - 中部显示阶段列表 `stageNo`
  - 每个阶段显示工时数、交付物历史、客户确认结果、管理员阶段审计结果
  - 底部显示最终总审计状态

### 3. 阶段模型与业务数据
- 为 `worklogs` 和 `deliverables` 新增显式 `stageNo`。
- `stageNo` 由后端自动管理，不让开发者手工选择：
  - 如果当前项目存在未通过阶段审计的阶段，新记录继续写入该阶段
  - 如果最近阶段已经通过管理员阶段审计，新记录自动进入 `nextStageNo`
- 客户确认继续以“交付物”为主体，但确认页按阶段展示：
  - 每个待确认阶段展示“该阶段最新交付物”
  - 同时展示该阶段全部工时历史
  - 同阶段旧交付物保留为历史记录
- 客户确认通过后，该阶段进入“待管理员阶段审计”。
- 客户确认驳回后，该阶段保持打开状态，开发者继续在同一 `stageNo` 下补工时/交付并再次提交。
- 管理员阶段审计通过后，该阶段关闭，后续新记录进入下一阶段。
- 管理员阶段审计驳回后，该阶段重新打开，继续使用同一 `stageNo` 修订。
- 新增统一的项目审计实体表，建议命名为 `project_audits`，记录：
  - `projectId`
  - `stageNo`（总审计为空）
  - `auditType`（`stage | final`）
  - `decision`（`pass | reject`）
  - `comment`
  - `txHash`
  - `actorId`
  - `createdAt`
- `evidence_records` 继续保留，并新增业务类型：
  - `stage_audit`
  - `final_audit`

### 4. 管理员审计页与客户端确认页
- 客户确认页从“待确认交付物”改为“待确认阶段”：
  - 展示阶段号
  - 展示该阶段最新交付物
  - 展示该阶段工时历史
  - 操作仍为“通过 / 驳回”
- 管理员审计页拆成两类工作队列：
  - `待阶段审计`
  - `待最终审计`
- 阶段审计弹窗提交参数固定为：
  - `projectId`
  - `stageNo`
  - `decision`
  - `comment`
- 最终总审计弹窗提交参数固定为：
  - `projectId`
  - `decision`
  - `comment`
- 最终总审计仅在以下条件满足时可用：
  - 不存在打开中的阶段
  - 至少有一个阶段已通过阶段审计
  - 项目当前状态仍为 `active`

### 5. 演示数据、页面与摘要联动
- bump 数据库种子版本，重建演示数据为“多阶段项目”：
  - 同一项目包含多个 `stageNo`
  - 至少一阶段已通过客户确认并完成管理员阶段审计
  - 至少一阶段仍在客户确认或管理员阶段审计中
- Dashboard、Audit、Process Flow 的摘要文案全部切换为阶段语义：
  - 待客户确认阶段
  - 待阶段审计阶段
  - 待最终审计项目
- README 与 `docs/release-*.md` 继续同步更新。

## Public Interfaces / Types
- `GET /api/users?role=<role>&status=<active|inactive|all>`：
  - 默认只返回 `active`
  - 返回项新增 `status`、`referencedProjectCount`、`canDelete`
- `POST /api/users`：管理员新增开发者/客户
- `PUT /api/users`：管理员更新成员资料
- `POST /api/users/password/reset`：管理员重置成员密码
- `POST /api/users/status`：管理员停用/启用成员
- `DELETE /api/users?id=<id>`：仅未被项目引用时允许删除
- `GET /api/projects`：项目返回保持不变，但成员选择只用 active 用户
- `GET /api/worklogs` / `GET /api/deliverables`：返回新增 `stageNo`
- `POST /api/worklogs` / `POST /api/deliverables`：请求体不新增字段，后端自动分配 `stageNo`
- `POST /api/evidence`：继续沿用现有演示回写；新增支持 `stage_audit` 和 `final_audit`
- `POST /api/audit/stage`：`{ projectId, stageNo, decision, comment }`
- `POST /api/audit/final`：改为 `{ projectId, decision, comment }`
- `GET /api/process-flow`：返回由线性 `projectSteps` 改为阶段循环结构，至少包含：
  - `globalSteps`
  - `selectedProject`
  - `stageSummaries[]`
  - `currentStage`
  - `finalAuditStatus`

## Test Plan
- 管理员可在“成员管理”页新增开发者和客户；新账号能登录，默认可被项目分配。
- 已引用成员执行删除时被阻止并提示只能停用；停用后不可登录、不可分配新项目，但历史项目仍可正常查看。
- 未引用成员允许删除，删除后不再出现在成员列表和项目下拉中。
- 项目管理页成员下拉仅显示 active 成员。
- 开发者在同一项目下连续登记多条工时、多次交付物时，后端会自动归入同一 `stageNo`，直到该阶段通过管理员阶段审计。
- 客户确认页能看到“当前阶段最新交付物 + 同阶段工时历史”；客户通过后生成待阶段审计队列。
- 管理员阶段审计通过后生成 `stage_audit` 演示链记录，并关闭该阶段；后续新工时/交付自动进入下一阶段。
- 管理员阶段审计驳回后，该阶段重新打开，开发者继续在同一 `stageNo` 下补记录。
- 最终总审计只能在所有阶段关闭后执行；通过后项目 `completed`，驳回后项目 `disputed`。
- 流程总览页能正确展示“准备阶段 + 阶段循环 + 最终总审计”的新模型。
- 前端构建、后端编译、演示数据重建、关键 API 联调全部通过。

## Assumptions / Defaults
- 独立成员管理页为 admin-only，位置固定在新路由 `/members`。
- 成员“完整管理”按“列表 + 新增 + 编辑资料 + 重置密码 + 停用/启用 + 条件删除”实现，不包含角色改动。
- `username` 和 `role` 创建后不允许修改。
- 阶段编号 `stageNo` 明确写入数据库，但由后端自动推进，不要求开发者手工选择。
- 客户确认保持“交付物为主、工时为辅”的模型，不引入单独的“批次审批”实体。
- 阶段审计与总审计继续使用演示链上回写，不扩展 Solidity 合约。
