# 流程总览重做方案

## Summary
- 本次按“两阶段交付”推进：先在仓库根目录新增 `test/process-flow-prototype/` 做独立原型，再在用户确认后接入正式的 `/process-flow` 页面。
- 正式页继续保留 `高层流程 / 阶段循环 / 当前快照` 三视图，不改路由与项目切换入口。
- 视觉方向锁定为“立体卡片地图 + 中等动态”：使用分层卡片、连线流动、hover 抬升、点击聚焦与展开过渡，不做夸张 3D 旋转或纯配色类优化。
- 阶段循环页重点改成“顶部紧凑角色条 + 当前阶段优先 + 历史阶段可展开查看”，明确展示开发者、客户、管理员各自卡在什么环节，以及最终总审计为何可做或不可做。

## Key Changes
- 第一阶段原型：
  - 在 `test/process-flow-prototype/` 放一个独立可打开的 HTML 原型，使用 mock 数据，不依赖后端接口。
  - 原型保留三视图切换，重点验证两块内容：高层流程的立体卡片地图、阶段循环的当前阶段重点卡 + 历史阶段内联展开。
  - 高层流程图采用固定拓扑：`项目创建 -> 钱包准备 -> 阶段登记 -> 客户确认 -> 管理员阶段审计 -> 最终总审计`，并额外表现“阶段审计未通过时回到阶段登记”的循环边。
  - 原型里的高层流程节点支持 hover 高亮、点击聚焦、当前节点脉冲、连线轻流动动画；阶段卡支持点击展开，展开后显示角色状态、前置条件、当前阻塞和下一步动作。
- 第二阶段正式接入：
  - `frontend/src/views/ProcessFlowView.vue` 重构为“同页三视图，但每个视图信息密度明显提升”。
  - 高层流程视图改成真正的流程图，不再是简单的纯文字步骤卡片；布局使用 3D 分层卡片地图，节点间用连线表现推进与回环。
  - 阶段循环视图取消现有左侧大角色栏，改为顶部紧凑角色条，三个角色只保留名称、当前状态、所在阶段、解锁条件摘要。
  - 阶段循环区把“当前阶段”固定放在最上方并默认展开，历史阶段按倒序列在下方，默认折叠，点击后内联展开详情。
  - 当前阶段卡必须明确展示：当前阶段编号、当前归属角色、开发者是否仍可登记、客户是否待确认、管理员是否待阶段审计、最终总审计是否仍锁定。
  - 当前快照视图改为更偏“门禁看板”：突出当前阶段状态、最终总审计前置条件、阻塞原因和通过链路，而不是只放两张摘要卡。
  - 共享样式继续放在现有全局样式体系里，不新增第三方动画库；使用 Vue 原生状态切换、CSS transform/transition、必要的内联 SVG 连线与 keyframes 完成动态效果。
- 页面交互规则：
  - 项目切换器继续保留在 Hero 区域，不改入口位置。
  - 默认进入 `阶段循环` 视图时，如果项目存在当前阶段，则自动聚焦并展开当前阶段。
  - 历史阶段允许查看，但不抢当前阶段视觉中心；当前阶段必须在结构、尺寸、信息量上明显大于历史阶段。
  - 最终总审计只在满足“所有阶段已关闭”时显示为可执行；否则必须直接显示当前阻塞原因，例如“等待客户确认阶段 N”或“等待管理员阶段审计阶段 N”。

## Public Interfaces
- `GET /api/process-flow` 保持现有字段不删除，但做增量扩展，前端正式接入时只消费新旧兼容结构。
- `globalSteps` 每项新增：`actor`、`state`、`detail`、`gate`、`isCurrent`。
- 新增 `roleProgress`：数组，固定包含 `admin / developer / client` 三项；每项包含 `role`、`title`、`stageNo`、`status`、`summary`、`nextAction`、`blockedBy`。
- `stageSummaries` 每项新增：`isCurrent`、`phase`、`currentOwner`、`requirement`、`nextAction`、`blockingReason`、`roleStates`。
- `finalAuditStatus` 新增：`blockingReason`、`blockingStageNo`、`requirements`。
- 这些字段由 `backend/src/server.c` 的 `/api/process-flow` 响应构造逻辑直接生成；字段命名固定，不在实现时再二次调整。

## Test Plan
- 原型验证：
  - 直接打开 `test/process-flow-prototype/` 页面，确认三视图都可切换。
  - 确认高层流程图存在立体层次、连线动效、节点聚焦和点击反馈，不是纯文字卡片。
  - 确认阶段循环里顶部角色条比现有角色栏更紧凑，但能直接看出开发者、客户、管理员当前所处环节。
  - 确认当前阶段默认展开，历史阶段可点击展开，展开信息包含角色进度、门禁条件和下一步动作。
- 接口验证：
  - 场景一：项目尚未进入任何阶段。
  - 场景二：当前阶段开放，开发者仍在登记。
  - 场景三：当前阶段交付已提交，等待客户确认。
  - 场景四：客户已确认，等待管理员阶段审计。
  - 场景五：所有阶段关闭，可执行最终总审计。
  - 场景六：项目已 `completed` 或 `disputed`。
- 正式页验证：
  - 管理员、开发者、客户三种角色分别进入 `/process-flow`，确认角色条文案和可见项目一致。
  - 切换不同项目时，流程图当前节点、当前阶段卡和总审计门禁状态会同步变化。
  - `npm run build` 通过；页面在桌面和小屏宽度下布局不溢出，历史阶段展开后不会破坏整体排布。

## Assumptions
- 原型阶段只做一个方向，不做多套候选；方向已经锁定为“立体卡片地图”。
- 原型目录固定放在仓库根目录 `test/process-flow-prototype/`，用于样式与交互确认，不接入正式路由。
- 正式集成前先以原型作为验收停点；原型未确认前，不继续做正式页面替换。
- 不引入新依赖，全部基于现有 Vue、CSS 和原生 SVG/CSS 动效实现。
- `/api/process-flow` 采用增量扩展而不是破坏性重命名，避免影响现有调用。
