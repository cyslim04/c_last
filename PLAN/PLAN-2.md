## 管理员项目管理页精简与跳转联动方案

### Summary
- 将管理员 `项目管理` 页的顶部总览改为 `PageHero` 的统一紧凑版，降低首屏占高，让筛选区和项目列表更早进入视野。
- 把 `进行中` 和 `已完成` 项目卡片改为整卡可点击，点击后跳转到 `流程总览`，并自动定位到对应项目。
- 不改后端接口；沿用现有 `/api/process-flow?projectId=...` 能力，仅补前端路由与页面状态同步。

### Key Changes
- `PageHero` 组件新增尺寸接口，采用 `size="compact"`（默认仍为 `default`），只在 `项目管理` 页启用。
- `compact` 视觉规范固定为：
  - 缩小 hero padding、标题字号、badge 间距、stats 卡片内边距。
  - 保留当前信息结构（标题、描述、badge、stats、右侧辅助卡），但整体高度明显低于默认版。
  - 默认版样式完全不变，其它页面无行为变化。
- `ProjectManagementView` 调整为：
  - 顶部 `PageHero` 切换到 compact 模式。
  - `active` / `completed` 项目渲染为整卡可点击入口，点击跳转到 `/process-flow?projectId=<id>`。
  - `disputed` / `draft` 项目保持静态卡片，不提供跳转。
  - 在“全部”分组视图和单状态筛选视图中，规则一致：只要卡片对应 `进行中` 或 `已完成`，就可点击。
  - 可点击卡片增加明确交互反馈：悬停态、指针样式、轻量文案提示或箭头指示，避免与静态卡片混淆。
- `ProcessFlowView` 调整为：
  - 接入 `useRoute`，在初次进入时读取 `route.query.projectId`。
  - 当 query 中有 `projectId` 时，调用现有 `api.getProcessFlow(projectId)`，让页面直接选中该项目。
  - 监听 query 变化；如果用户在同页再次通过路由切换项目，页面同步刷新选中项目。
  - 保持当前默认视图模式为 `阶段循环`，不额外引入新的 view 参数。
  - 若 query 中项目无效或当前角色不可见，前端不额外报错，沿用后端现有回退逻辑，落到默认可见项目。

### Public Interfaces
- 组件接口新增：`PageHero.size: "default" | "compact"`，默认 `"default"`。
- 页面路由约定新增并固定：`/process-flow?projectId=<projectId>` 用于指定流程总览初始选中项目。
- 后端接口、返回结构、数据库表均不修改。

### Test Plan
- 构建验证：`frontend` 执行 `npm run build` 通过。
- 管理员进入 `项目管理` 页：
  - 首屏 hero 明显更紧凑，筛选区与项目列表无需大幅滚动即可看到。
  - `进行中`、`已完成` 卡片为整卡可点，`争议中`、`草稿` 不可点。
- 在“全部”视图下点击一张 `进行中` 卡片：
  - 跳到 `流程总览`。
  - 顶部项目切换器显示被点击项目。
  - 页面内容与该项目阶段数据一致。
- 在“已完成”筛选下点击一张已完成卡片：
  - 同样跳到 `流程总览` 并定位到该项目。
- 直接访问或刷新 `/process-flow?projectId=<id>`：
  - 仍能正确选中对应项目。
- 访问无效或无权限的 `projectId`：
  - 页面不崩溃，回退到后端返回的默认可见项目。
- 响应式检查：
  - `compact` hero 在桌面与移动宽度下都不出现排版重叠或按钮挤压。

### Assumptions
- “点击已完成或进行中项目”按整张项目卡片处理，而不是仅点击状态标签。
- “总览屏缩小一点”按通用组件能力实现，但本次只在 `项目管理` 页落地启用。
- 跳转后的目标页面保持当前既有默认视图 `阶段循环`，不新增其它导航层级。
