# 管理后台头部、登录快捷登录与最终审计修复方案

## Summary
- 左上角信息区改为紧凑身份信息，只保留 `角色 + 姓名 + 当前状态`，移除冗余内容，保持头部整体布局不变。
- 登录页仅替换“快捷登录”区域：改为 `管理员 / 开发者 / 客户` 三个下拉框，收起态保持简洁，展开态显示详细信息，选中后自动填充账号和密码，但不自动提交。
- 管理员“审计管理”页恢复显式 `最终审计` 主按钮，放在页面操作区可直接看到的位置，不再隐藏。

## Key Changes
- 头部信息区
  - 将现有左上角展示统一为身份块：第一行显示角色徽标与姓名，第二行显示会话状态。
  - 无真实姓名时回退显示账号名，状态默认显示 `已登录`。
  - 不调整其他头部区域，只优化左上角文案、层级和间距。
- 登录页快捷登录
  - 将快捷登录数据改为三类分组：`admin[] / developer[] / customer[]`。
  - 每个选项固定包含：`id、displayName、username、password、description`。
  - 页面渲染三个独立下拉框；任一时刻只保留一个有效选择，切换分类时清空其他两个下拉的已选项。
  - 下拉展开项展示 `姓名 + 账号 + 密码 + 说明` 的多行详细信息；收起态只显示简短标签，避免登录页显得拥挤。
  - 选择后自动写入账号、密码输入框，但保留手动修改能力；登录页其他说明、文案和布局不改。
- 管理员最终审计
  - 在“审计管理”页顶部操作区增加明确的 `最终审计` 主按钮，不放入更多菜单，不依赖 hover 才出现。
  - 按钮默认禁用，选中一条待最终审计记录后启用；未选中时给出明确提示，例如“请选择待审记录”。
  - 点击按钮打开审计弹窗，展示当前记录关键信息，并提供 `通过 / 驳回 / 取消` 操作。
  - 提交时使用 `targetId、decision、comment`；提交成功后刷新列表、更新状态，并同步按钮可用性。
  - 最终审计入口和弹窗仅管理员可见，开发者和客户端不显示。

## Public Interfaces / Types
- 快捷登录前端配置结构调整为按角色分组的数据结构：`admin[] / developer[] / customer[]`。
- 快捷登录项字段统一为：`id、displayName、username、password、description`。
- 最终审计提交动作统一为：`targetId、decision(pass|reject)、comment`。
- 登录接口和其他页面数据契约不改；审计接口优先复用现有命名和返回结构。

## Test Plan
- 管理员、开发者、客户三种登录态下，左上角都正确显示 `角色 + 姓名 + 已登录`，无姓名时回退账号名。
- 登录页除快捷登录区域外，无布局和文案回归。
- 三个下拉框收起态保持简洁，展开态能看到 `姓名 + 账号 + 密码 + 说明`。
- 选择任一快捷登录项后，账号和密码自动填充，但不会自动提交登录。
- 切换不同角色下拉时，旧选择被清除，表单值以最后一次选择为准。
- 管理员进入审计管理页时可直接看到 `最终审计` 按钮；未选中记录时禁用，选中待审记录后可点击。
- 最终审计弹窗可执行通过和驳回，提交后列表刷新、状态更新。
- 非管理员角色看不到最终审计相关入口。

## Assumptions
- “左上角信息合理一点”按 `角色 + 姓名 + 当前登录状态` 处理，不保留其他辅助字段。
- “在下拉框中显示详细信息”指详细信息只在下拉展开层中展示，不常驻登录页主视图。
- “最终审计”是对待审记录执行的最终审核动作，因此页面主按钮依赖当前所选记录，不单独拆成新导航页。
- 本轮本地 shell 无法读取仓库目录，具体文件落点需按现有前端组件、路由和状态管理约定接入；以上行为定义即为实施标准。
