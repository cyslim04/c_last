# Trust Work 两周工程化重构计划

## 摘要
- 目标：把当前“演示可用”的实现收敛成“可继续演进”的工程结构，优先解决后端单文件过重、前端页面与接口耦合、自动化验证薄弱三类问题。
- 原则：不改现有业务闭环、不改接口路径、不改数据库实体语义、不改合约 ABI；允许少量低风险体验微调，但不做新功能和视觉重设计。
- 交付结果：后端按领域拆分、前端按数据层与页面层拆分、补齐最小测试门禁、同步工程文档与版本日志。

## 实施安排
### 第 1 周
1. 基线与护栏（第 1 天）
- 固化当前回归基线：登录、快捷账号、项目管理、工时登记、交付物、客户确认、阶段审计、最终审计、流程总览。
- 记录当前构建与运行命令，整理一份“重构前行为清单”，后续所有改动都对照回归。
- 锁定本轮边界：不改接口 URL、不改会话存储 key、不改数据库表名字段名、不改合约对外函数名。

2. 后端拆分（第 2-4 天）
- 将后端按职责拆成 `main/router`、`http`、`db`、`auth/users`、`projects/worklogs/deliverables`、`audit/process_flow` 六类模块，`main` 只负责启动与生命周期。
- 保留原有 REST 路径与业务行为，把路由层改成集中式方法+路径分发，停止继续扩张单个总文件。
- 提炼公共能力：请求解析、JSON 响应、错误响应、认证校验、SQLite 结果读取、密码哈希、token 生成。
- 统一错误响应格式为 `{ message, code? }`；成功响应保持兼容，不引入需要前端联调的新字段。
- 本轮不做数据库 schema 重设计，不新增迁移逻辑；仅允许新增内部 helper、常量和头文件组织。

3. 前端数据层收敛（第 5 天）
- 将前端接口层拆成 `client` + 按领域划分的 API 模块，覆盖 `auth/users/projects/worklogs/deliverables/audit/processFlow`。
- 保持现有 Pinia store 数量和职责不变，本轮不新增全局状态中心。
- 统一请求异常抛出与页面消费方式，避免每个页面自行拼装错误处理逻辑。

### 第 2 周
4. 前端页面解耦（第 6-7 天）
- 重点拆分三个最重页面：流程总览、审计管理、成员管理；页面文件只保留数据编排和事件响应，展示区块下沉为子组件。
- 抽出通用异步状态模式：`loading / empty / error / ready`，用于列表页和流程页，减少重复状态分支。
- 保持现有路由、导航结构和主题体系；体验微调只做状态呈现统一与稳定性修正，不改页面信息架构。

5. 质量门禁（第 8-9 天）
- 前端新增轻量测试依赖：`Vitest` + `@vue/test-utils`，覆盖会话恢复、路由守卫、至少一个重页面的状态切换逻辑。
- 后端新增 PowerShell smoke 脚本，串起 `登录 -> 取项目 -> 开发者登记 -> 客户确认 -> 管理员阶段审计/最终审计` 的最小闭环，并覆盖未授权访问失败场景。
- 合约继续沿用 Hardhat，补充边界测试：非法钱包、无效时间段、重复记录、越权确认、读取不存在证据。
- 质量门禁固定为：前端 `build + test`，后端 `build + smoke`，合约 `compile + test`。

6. 文档与收口（第 10 天）
- 补一份工程说明，明确后端模块图、前端数据流、测试命令和回归入口。
- 同步 `docs/reference` 中 API 和数据库说明，只修正文档与现状不一致之处，不重写业务说明书。
- 按仓库规则写 5 篇里程碑日志：后端基线、后端拆分、前端数据层、前端页面层、测试与文档；第 5 篇后同步更新 README 摘要。

## 公共接口与类型约束
- REST 接口路径、请求体、成功响应主体保持兼容；本轮不新增必须由前端配合的新接口字段。
- 后端错误响应统一为 `{ message, code? }`，其中 `code` 为可选辅助字段，前端不依赖该字段也能正常工作。
- 前端保留现有本地会话存储 key `trust-work-session` 和现有 store 对外接口。
- 合约 ABI、部署脚本使用方式和链上业务模型本轮不变。

## 测试与验收
- 管理员、开发者、客户三角色登录与路由守卫行为与当前一致。
- 快捷登录、成员管理、项目管理、工时、交付物、客户确认、阶段审计、最终审计、流程总览全部至少完成一次回归。
- 后端拆分后可重复构建、重复启动，不出现接口路径回归、认证失效或角色权限错乱。
- 前端重页面拆分后构建通过，关键页面的 `loading / empty / error` 状态展示一致。
- 合约测试全部通过，新增边界测试覆盖当前已知业务约束。
- 文档、README、版本日志全部符合中文与日志归档规范。

## 默认假设
- 本轮是重构，不是重写；不引入新后端框架，不迁移数据库，不改链上业务模型。
- 允许少量体验微调，但仅限异步状态、错误提示和页面稳定性，不做新的视觉方案。
- 新增工具仅限前端轻量测试依赖；后端 smoke 优先使用现有 Windows/PowerShell 运行环境。
- 若重构中发现历史行为与文档不一致，以当前可运行、可演示的行为为准，先保持兼容，再在日志中注明。
