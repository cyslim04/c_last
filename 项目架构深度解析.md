# Trust Work 平台 —— 项目架构深度解析

## 一、项目概述

**Trust Work**（可信工时与交付存证平台）是一套面向毕业设计答辩场景的区块链存证演示系统。平台以 **管理员 - 开发者 - 客户** 三角色协作为核心，实现了一套基于 SHA-256 哈希的"工时登记 → 交付物提交 → 客户确认 → 管理员审计"阶段循环式工作流，并具备链上存证回写能力。

系统由三大部分组成：

| 层级 | 技术选型 | 定位 |
|------|---------|------|
| 后端 | C (C11) + Winsock2 + SQLite3 | 单体 HTTP 服务器，零框架依赖 |
| 前端 | Vue 3 (Composition API) + Pinia + Vite | SPA 单页应用，含多角色快捷登录与流程展示 |
| 合约 | Solidity 0.8.24 + Hardhat | 以太坊智能合约，链上证据存证 |

### 核心数据流

```
1. 用户登录 → POST /api/auth/login → SQLite 验证 → 返回 Bearer Token
2. 前端请求 → api/http.js 注入 Token → 后端 route_request() → 业务处理器 → SQLite → JSON 响应
3. 开发者登记工时 → POST /api/worklogs → SHA-256 摘要生成 → 写入 SQLite → 返回
4. 链上存证 → web3/evidence.js → MetaMask 签名 → TrustWorkProof.sol → 写回 POST /api/evidence
5. 客户确认 → POST /api/evidence (deliverable_confirmation) → SQLite 写入审批记录
6. 管理员审计 → POST /api/audit/stage || /audit/final → 项目状态流转
```

### 5.2 HTTP 服务器实现

```c
// 核心模式：单线程同步阻塞
main() {
  WSAStartup();                      // 初始化 Winsock
  init_database();                   // 初始化 SQLite + 建表 + 种子数据
  socket() → bind() → listen();      // 创建 TCP 服务 (端口 8080)
  while (1) {
    SOCKET client = accept();        // 阻塞等待连接
    recv_request(client, &request);  // 解析 HTTP 请求
    route_request(&app, client, &request);  // 路由分发
    closesocket(client);             // 关闭连接
  }
}
```

- 无并发模型：同一时间仅处理一个请求
- 无 HTTP 库依赖：手动解析 `Content-Length`, `Content-Type`, `Authorization` 头
- CORS 支持：每个响应中注入 `Access-Control-Allow-Origin: *` 和相关头
- 支持 OPTIONS 预检：直接返回 200

### 5.3 JSON 解析（自研）

不使用任何 JSON 库，而是通过纯字符串搜索实现：

```c
// 从原始 JSON 字符串中提取指定 key 的字符串值
json_get_string(json, "username", output, sizeof(output));

// 从原始 JSON 字符串中提取指定 key 的整数值
json_get_int(json, "projectId", &value);

// 对输出进行 JSON 转义（处理 "、\、\n）
json_escape(input, escaped_output, sizeof(escaped_output));
```

实现方式为 `strstr` 搜索 `"key"` 模式 + 手写状态机解析值。这种方式简洁但脆弱——不处理嵌套对象、数组、Unicode 转义等复杂场景，但对于当前 API 的扁平 JSON 结构足够。

### 5.4 认证体系

**密码存储**：`SHA256(password:salt)`，salt 格式为 `{prefix}-{username}`
- 种子账号的 salt 前缀为 `"seed"`
- 用户自注册账号的 salt 前缀为 `"user"`

**Token 机制**（简化版）：
- 登录成功返回 `{ token: "demo-token-{userId}", user: {...} }`
- Token 不是 JWT——只是字面字符串 `"demo-token-{userId}"`
- 验证时解析 `userId` 从 SQLite 查询用户信息
- 会话存储在浏览器 `localStorage` 中

**关键认证代码**：
```c
// 解析 Bearer token 格式
static int parse_auth_token(const HttpRequest *request, int *user_id) {
  const char *prefix = "Bearer demo-token-";
  // 从 Authorization 头中提取 userId
  sscanf(token + strlen(prefix), "%d", user_id);
}
```

### 5.7 Schema 迁移策略

`ensure_schema()` 函数采用 **累积式 ALTER TABLE** 迁移模式（第 764-777 行）：

```c
// 先执行完整 CREATE TABLE IF NOT EXISTS
exec_sql(db, schema);
// 再用 ALTER TABLE 逐个补偿历史版本缺失的列
exec_sql_ignore(db, "ALTER TABLE users ADD COLUMN email TEXT DEFAULT ''");
exec_sql_ignore(db, "ALTER TABLE users ADD COLUMN phone TEXT DEFAULT ''");
// ... 更多列迁移
```

`exec_sql_ignore` 在 ALTER 失败时（列已存在）静默忽略错误。这是 SQLite 无原生 `ADD COLUMN IF NOT EXISTS` 的折中方案。

## 七、智能合约深度解析

### 7.1 TrustWorkProof.sol

合约约 120 行，是轻量级证据存证合约：

```solidity
contract TrustWorkProof {
    // 项目结构
    struct Project {
        uint256 projectId;
        address developerWallet;
        address clientWallet;
        bool exists;
    }

    // 证据记录结构
    struct EvidenceRecord {
        uint256 projectId;
        uint256 recordId;
        bytes32 contentHash;      // 证据内容的 SHA-256 哈希
        uint64 timestamp;
        address submitter;
        string recordType;         // worklog / 版本号
        bool confirmed;
    }
}
```

**核心函数**：

| 函数 | 访问控制 | 功能 |
|------|---------|------|
| `createProject(id, devAddr, clientAddr)` | 公开 | 注册项目，建立开发者/客户钱包映射 |
| `recordWorklog(projectId, worklogId, hash, start, end)` | `onlyDeveloper` | 开发者在链上记录工时，自动 `confirmed = true` |
| `recordDeliverable(projectId, deliverableId, hash, version, nameHash)` | `onlyDeveloper` | 开发者上传交付物，初始 `confirmed = false` |
| `confirmDeliverable(projectId, deliverableId, approved, commentHash)` | `onlyClient` | 客户确认/驳回交付物 |
| `getEvidence(projectId, recordId)` | 公开 view | 查询证据记录 |

**访问控制**：通过 `onlyDeveloper` 和 `onlyClient` 修饰器检查 `msg.sender` 是否与注册的钱包地址一致。

**事件系统**：
- `ProjectCreated` — 项目创建
- `WorklogRecorded` — 工时记录（含哈希和时间范围）
- `DeliverableRecorded` — 交付物记录（含文件哈希、版本、文件名哈希）
- `DeliverableConfirmed` — 交付物确认（含审批结果和评论哈希）

## 八、核心业务流程

### 8.1 阶段循环式工作流

这是平台最核心的业务模型：

```
┌──────────────────────────────────────────────────────────────┐
│                    项目生命周期 (Project)                      │
│                                                              │
│  [draft] → [active] → [completed] 或 [disputed]              │
│                                                              │
│  active 状态下进入阶段循环:                                     │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐    │
│  │               阶段 N (Stage N)                        │    │
│  │                                                      │    │
│  │  1. [开发者]  登记工时  (worklogs)                     │    │
│  │     │                                                │    │
│  │  2. [开发者]  提交交付物 (deliverables)                │    │
│  │     │                                                │    │
│  │  3. [开发者]  发起链上回写 (evidence)                  │    │
│  │     │                                                │    │
│  │  4. [客户]    阶段确认 (confirm deliverable)          │    │
│  │     │                                                │    │
│  │  5. [管理员]  阶段审计 (stage audit: pass/reject)      │    │
│  │     │                                                │    │
│  │     ├─ pass → 进入阶段 N+1（回到步骤 1）               │    │
│  │     └─ reject → 阶段保持 open，开发者继续修改          │    │
│  └──────────────────────────────────────────────────────┘    │
│                          │                                   │
│          所有阶段均 pass 后 ↓                                  │
│                          │                                   │
│  6. [管理员]  最终总审计 (final audit)                        │
│     ├─ pass → project.status = "completed"                   │
│     └─ reject → project.status = "disputed"                  │
└──────────────────────────────────────────────────────────────┘
```

### 8.2 阶段状态计算（后端核心逻辑）

```c
// StageSnapshot.state 的三种状态
// "open" — 开发者可以继续登记
// "pending_stage_audit" — 等待管理员阶段审计
// "closed" — 阶段已完成审计，锁定

// 阶段推进逻辑 project_current_stage_for_write()
// - 最近阶段 closed → 新记录进入 stage_no + 1
// - 最近阶段 open → 新记录继续进入当前 stage_no
// - 最近阶段 pending_stage_audit → 禁止登记
```

### 8.3 关键状态流转表

| 当前状态 | 触发操作 | 目标状态 | 角色 |
|---------|---------|---------|------|
| 项目 draft | 创建完成 | active | Admin |
| 阶段 open | 提交交付物 | open (累计) | Developer |
| 阶段 open | 客户确认 | pending_stage_audit | Client |
| pending_stage_audit | 管理员阶段审计 pass | closed (进入下一阶段) | Admin |
| pending_stage_audit | 管理员阶段审计 reject | open (保持当前阶段) | Admin |
| 所有阶段 closed | 管理员最终审计 pass | active → completed | Admin |
| 所有阶段 closed | 管理员最终审计 reject | active → disputed | Admin |

## 十、SHA-256 应用全景

SHA-256 在该平台中被应用于四个核心场景：

### 10.1 密码哈希

```
hash = SHA256(password:salt)
salt = "{seed|user}-{username}"
```

- 种子账号 salt: `seed-admin`, `seed-dev` 等
- 用户创建账号 salt: `user-{username}`
- 这是简化版的加盐哈希，不具备 bcrypt/argon2 的工作量因子

### 10.2 工时摘要

```
digest = SHA256(JSON.stringify({
  projectId, userId, stageNo, workDate, hours, taskDescription
}))
```

生成规范 JSON 的 SHA-256 摘要，用于工时记录的不可篡改性验证。

### 10.3 交付物文件哈希

```javascript
// 浏览器端通过 Web Crypto API 计算
const buffer = await file.arrayBuffer();
const hashBuffer = await crypto.subtle.digest("SHA-256", buffer);
const hexHash = Array.from(new Uint8Array(hashBuffer))
  .map(b => b.toString(16).padStart(2, "0"))
  .join("");
```

上传文件时在浏览器端计算 SHA-256 哈希值，作为文件的数字指纹。

### 10.4 区块链证据哈希

```solidity
// 合约中使用 bytes32 存储 SHA-256 哈希
function recordWorklog(
    uint256 projectId,
    uint256 worklogId,
    bytes32 worklogHash,  // SHA-256 摘要
    uint64 periodStart,
    uint64 periodEnd
) external onlyDeveloper(projectId) { ... }
```

前端生成摘要后通过 Ethers.js 写入合约，实现链上不可篡改存证。

## 十一、架构设计评价

### 11.1 设计优点

1. **极简依赖**：后端零框架，前端仅依赖 Vue + Pinia + Router + Ethers.js 四个核心库，易于理解和教学
2. **单体文件清晰**：`server.c` 虽大（3592 行），但逻辑组织有章可循，函数单一职责
3. **演示友好**：一键启动、种子数据、快捷登录和流程总览，为答辩场景做了充分准备
4. **哈希防线**：SHA-256 贯穿密码认证、工时存证、文件指纹、链上记录四个关键环节
5. **阶段建模完整**：从工时登记到最终审计形成闭环，状态机设计合理
6. **双主题支持**：暗色/亮色主题通过 CSS 变量优雅切换，提升了演示视觉效果

### 11.2 已知局限与改进方向

1. **并发能力**：单线程同步模型无法处理并发请求，适合演示和教学但不适合生产
2. **Token 安全性**：`demo-token-{userId}` 模式无签名无过期机制，仅适合演示
3. **JSON 解析脆弱**：自研解析器基于字符串搜索，不处理嵌套、转义、数组等复杂场景
4. **密码安全**：SHA-256 + 短盐仅满足演示需要，生产环境应使用 bcrypt/argon2
5. **无自动化测试**：前后端缺乏自动化测试套件，验证依赖手动操作
6. **平台绑定**：后端使用 Winsock2，仅支持 Windows
7. **SQL 注入风险**：部分 SQL 拼接未充分参数化
8. **链上交互不完整**：多数操作仍为前端生成 mock tx_hash 写入后端，非全量链上交易
9. **种子数据硬编码**：演示数据以 C 代码形式硬编码，不易维护
10. **无 HTTPS**：HTTP 明文传输，Token 可被中间人截获
